What Numara does

Most net-worth tools fail because they demand structured data entry: people give up because the friction of typing balances from twelve different statements is too high. Numara inverts that. The primary input is a photo or upload of a statement, and the system figures out what the document represents, what the balance is, and whether it is new or an update to an existing account.

Outputs are intentionally simple: a single dashboard of total net worth in USD, CAD, and GBP, broken down by category — real estate, investments, cash, liabilities, recurring outgoings — with the ability to drill into any asset for detail and, for real estate, a one-tap "what would I net if I sold this today?" view.

Built for one user initially (the author) but structured so it can extend to a household view later. The data model is household-ready from day one; the v1 UI is single-user.

The PRD lays out a 12-step build sequence. Each step is a discrete shippable unit. All twelve are now landed in code. Some require one-off configuration before they exercise end-to-end (mainly the AI prompts).

4Document upload done

documents table. Private Supabase Storage bucket, household-scoped prefix RLS. 15-min signed URLs. Standalone /capture + attach-to-balance.

5Admin foundation done

Eight admin tables (secrets, settings, models, prompts, versions, bindings, call logs, audit log). AES-256-GCM encryption layer. OpenRouter /models sync. Admin Settings page. llmCall() service with primary + 2 fallback chain.

6Admin Prompts page done

List + editor with smart-diff save (binding-only changes don't bump body version). Searchable virtualised model picker reused across 3 slots. Per-actor test rate limit. Full version history with one-click revert.

7AI extraction code done · prompt to enable

llmCall + multimodal image_url support. extract_statement runs synchronously on upload. Review form with editable fields + per-field confidence pills. Falls back to manual entry on failure.

8New-vs-update classification done

Deterministic match (institution + last4 + currency) handles common cases. classify_new_vs_update is the LLM tiebreaker when ambiguous. Proposal banner + asset picker pre-selection.

9Real estate done

properties + property_valuations. Net-equity headline. Property panel with mortgage linkage. Sale-scenario calculator with jurisdiction defaults (1.25% UK, 5% CA, 6% US agent fees). CGT defaults off per PRD.

10PWA + offline done

Manifest, icons (capital N via next/og), iOS meta tags. Service worker via Serwist with NetworkFirst for HTML / CacheFirst for static. IndexedDB queue for offline captures with on-reconnect drain. Offline banner.

11Privacy + staleness + alerts done

Cookie-based privacy mode masks every figure server-side. 90-day staleness threshold. Amber alerts strip lists stale assets with deep links.

12History chart done

net_worth_snapshots. Computation walks assets, takes latest-on-or-before, applies FX of that day, pre-computes totals in each display currency. Hand-rolled SVG line chart. Manual "Snapshot now" button.

Architecture at a glance

Everything runs in Next.js's App Router. The proxy (formerly middleware) gates all routes by Supabase session. Supabase Postgres holds the data; row-level security policies enforce household scoping on every read and write. Storage holds original documents in a private bucket with the same RLS model. OpenRouter is the single outbound LLM gateway — no provider SDKs are imported directly. frankfurter.app supplies daily FX.

Five-box architecture. Solid arrows are request flow; the trust-boundary box lists the invariants enforced across all routes.

Capture flow — the headline feature

This is the path the PRD inverts the market around. Every other "track your net worth" app makes you type numbers from a statement. Numara has you take a photo and confirm.

Steps 1–7 of the capture flow. Online and offline paths converge at step 3 — the API endpoint is the same whichever way you arrived.

Auth flow

Magic-link via Supabase Auth. Passkeys/WebAuthn is the documented primary in PRD §6.1 but landed as a follow-up (see "What's next"). The signup trigger means a brand-new account never sees an empty users table — household and profile row are provisioned atomically with the auth.users insert.

Magic-link sign-in. The trigger pattern means there is no "if no profile, create one" code path — Postgres does it.

Data model

Two trust domains. Household-scoped tables hold user data. Global tables hold AI infrastructure that is shared across all households (currently just the one) and is only readable by users with is_system_admin = true.

Solid arrows are required FKs. Dashed arrows are nullable FKs or join keys. Property valuations link back to a synthetic balance_entries row so the dashboard math stays unified.

AI / LLM layer

The PRD made one structural decision that's load-bearing for everything downstream: all LLM calls go through OpenRouter, never a provider SDK. That keeps models swappable, lets each prompt have a primary + 2 fallback chain, and gives us one chokepoint for keys, logging, and cost control. The application code calls llmCall(slug, vars); everything else is configured in the admin UI.

Single chokepoint architecture. The runtime tries up to three models per call; the admin UI is the only place model selection or prompt body lives.

Offline / PWA

Service worker registered on first load (production builds only — disabled in dev so HMR works). NetworkFirst for HTML routes means a navigation falls back to the last cached version when offline. The IndexedDB upload queue handles the only write path the PRD explicitly calls out for offline: captures.

Online and offline both terminate at the same POST endpoint; the queue is just a deferred replay of what the online path does immediately.

Dashboard composition

Server-rendered, no client-side data fetching on the main load. Every section reads the same household state via the supabase server client.

Five sections, all server-rendered from a single round-trip to Supabase.

How money figures get to the screen

  1. Server fetches non-archived assets via RLS (only your household's rows are visible).
  2. Server fetches asset_latest_balances view to get the latest balance per asset in one query.
  3. Server reads display_currency + privacy_mode cookies.
  4. ensureFreshRates() hits frankfurter once a day, otherwise uses cached fx_rates.
  5. For each asset, convert(amount, native, display, rates) computes the display-currency figure. Liabilities subtract.
  6. Server renders all figures via formatMoney(n, ccy, masked) — when masked is true the digits never leave the server.
  7. Per-category subtotals and the headline net worth are sums of these converted figures.
  8. The history chart reads pre-computed snapshots — no recomputation at render time.

RLS guarantees verified

  • RLS enabled on every household-scoped table and on storage objects in the documents bucket.
  • current_household_id() SECURITY DEFINER helper used both as column default and inside RLS predicates → impossible to fabricate cross-household writes from the wire.
  • Storage bucket keys start with <household_id>/ and the storage RLS policy validates that prefix.
  • Service-role key never reaches the client (import 'server-only' guard on lib/supabase/admin.ts).
  • OpenRouter API key encrypted at rest with a key separate from the Supabase keys.

What needs one-off configuration

None of these are bugs. They are configuration steps the application UI was built for. You are the admin and you are expected to do these once.

1. Configure the extract_statement prompt needed for AI

Until this is set, every document upload will fall through to manual entry. The application code is ready; the prompt body and bound model are not.

Go to /admin/prompts/extract_statement and:

  1. Paste a real prompt body (the recommended body is in the chat history of the session that built step 7). It should ask the model to return a strict JSON shape with confidences per field.
  2. Set Available slugs to: known_institutions, user_default_currency, document_image_or_text.
  3. Pick a vision-capable model as the primary (search "claude sonnet" in the picker).
  4. Set fallbacks: openai/gpt-4o (or gpt-4.1), and a Gemini vision model.
  5. Temperature 0, max tokens 2048, response format JSON with the schema pasted in.
  6. Click Save, then Enable at the top.

2. Configure the classify_new_vs_update prompt optional

Only invoked as a tiebreaker when deterministic matching has more than one candidate. Most uploads never need it. Configure when you start hitting ambiguity.

3. Auth: passkey enrolment deferred follow-up

PRD §6.1 lists passkey/WebAuthn as the primary auth, with magic link as the fallback. We agreed at step 1 to ship magic link and add passkey as a dedicated follow-up. Magic link works today; passkey enrolment needs SimpleWebAuthn + Supabase Auth integration.

4. Vercel deploy + monthly snapshot cron deferred

Not deployed yet. When you deploy:

  • Add the .env.local values to Vercel project env (anon key, service role, OpenRouter key, encryption key).
  • Add a vercel.json cron entry pointing at a new /api/cron/monthly-snapshot route handler that iterates households and calls computeAndStoreSnapshot().
  • Configure Supabase Auth redirect URLs to include your production hostname.

5. OpenRouter cost dashboard deferred

The Settings page shows "recent calls" but not the aggregated cost-by-prompt breakdown the PRD describes. Data is all in llm_call_logs; needs a query and a small chart.

6. CSV / XLSX extraction deferred

The upload bucket accepts these MIME types, but lib/extraction.ts only ships images and PDFs to the vision model. CSV/XLSX would need text extraction first.

7. Recurring outgoings deferred

PRD §5.8 (cash-flow view) is a separate feature not in the §13 build sequence. The data model is in PRD §8; the UI and ingestion are unbuilt.

What to do next

If you want to start using it

  1. Configure extract_statement as above. Without this, AI extraction is a no-op.
  2. Add a couple of cash assets manually so the dashboard isn't empty.
  3. Take a real photo of a statement, upload it via /capture, watch the review form pre-fill.
  4. Click Snapshot now a few times across different days to start a history.

If you want to ship it

  1. Deploy to Vercel. Set env vars. Configure Supabase Auth redirect URLs.
  2. Add the monthly snapshot cron. The function exists; just needs a route handler + vercel.json.
  3. Decide on the passkey story. Either ship magic-link only or add SimpleWebAuthn before launching to anyone else.
  4. Run through the data-deletion path manually — PRD §6.1 promises export + 30-day grace deletion, which we haven't tested.

If you want to extend the AI

  1. Wire classify_new_vs_update (the prompt exists, the application call site exists; just configure the body and a model).
  2. Add the cost dashboard on /admin/settings — query llm_call_logs grouped by prompt_slug, ranked by spend, with a fallback-rate column.
  3. Consider OCR fallback (Tesseract / a hosted OCR API) for very simple documents to reduce token spend, as PRD §11 risk-mitigation suggests.
